S SafePass Mobile Design Guide

The web app's design, ready for mobile

Every value here comes from the shipped SafePass web app - the DM Sans type, the color palette, the borderless shadowed cards, the navy chrome. Build with these tokens and the mobile apps read as the same product. Sample screens are flavored for the admin/management and mapping/site-configuration apps; they're references, not pixel specs. Toggle the theme (top right) - both modes ship today. Questions or gaps: ask the SafePass web team.

Principles

When a case isn't covered by a specific rule, fall back to these.

1Feedback is immediateEvery action shows visible feedback within 100 ms - loading states on buttons, a flash on success, inline errors on failure.
2Progressive disclosureShow only what's needed at each step. Summary → drill-down; sheets and modals for secondary actions.
3Consistency over noveltyTwo surfaces that do similar things look and behave the same. Reuse the established pattern.
4Destructive actions confirmDelete, revoke, disconnect - always an explicit confirmation naming the specific resource.
5Errors are actionableSay what happened and what to do about it. Surface the API's error detail when available.
6Concise by defaultChatty UI is a bug. Status words over sentences. No captions on self-evident controls.

Typography & shape

DM Sans (400 / 500 / 700) everywhere - bundle it in the apps. Headings: weight 600, line-height 1.2, in the heading color. Corners are square-ish: 4px on nearly everything, 8px on modals and sheets. Elevation is a barely-there blue-tinted shadow - cards have no border.

Rules digest for mobile

The web standards translated to app terms. Accessibility baseline is WCAG 2.1 AA: 4.5:1 text contrast, visible focus, labels on every input, never color alone for meaning, respect reduced-motion.

Do

  • Bundle DM Sans; headings 600 in the heading color, body in slate.
  • Use the palette exactly - primary #3378FF, danger #C03221, the status map, the scope colors. Same level = same color in every app.
  • Keep the surface recipe: borderless cards, soft blue-tinted shadow, 4px corners, off-white body behind white cards.
  • Carry the navy #0C2556 chrome into navigation.
  • Toast for action results; inline errors for validation; banner alerts for screen-level errors.
  • Descriptive loading states and specific empty states, always.
  • Confirm every destructive action, naming the resource.
  • Display backend state as-is; support light and dark from day one.

Do not

  • Hardcode hex values at call sites - define tokens once and reference them.
  • Make badges pills or buttons bold - badges are 4px rectangles, buttons weight 400.
  • Put borders on cards or heavy shadows anywhere - elevation is whisper-quiet.
  • Use a modal for success confirmation - that's a toast. Or toast a validation error - that's inline.
  • Show a blank screen while loading, or "no data" before data arrives.
  • Map status → color ad hoc per screen - one shared helper.
  • Transform, rename, or re-derive backend states client-side.
  • Caption self-evident controls or restate a card's title in its body.
  • Invent a new pattern when an established one exists.
